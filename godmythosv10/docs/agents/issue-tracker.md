## Issue templates

### Bug report

```markdown
## What's broken

{One-paragraph description of the symptom from the user's perspective.}

## Expected behavior

{What should happen instead.}

## Steps to reproduce

1. ...
2. ...
3. ...

## Environment

- {OS / runtime / version}
- {Browser / client if relevant}
- {Commit SHA or release version}

## Additional context

{Logs, screenshots, related issues, etc.}
```

### Enhancement / feature request

```markdown
## Problem statement

{The problem from the user's perspective. What can they not do today?}

## Proposed solution

{High-level. What would the user-visible behavior be?}

## Alternatives considered

{Other approaches and why this one is preferred.}

## Out of scope

{What this enhancement explicitly does NOT do.}
```

### Vertical slice (from `TO_ISSUES`)

See `references/agent-brief.md` in the godmythos skill for the full agent
brief format. Each slice issue carries:

- Parent reference (PRD or umbrella issue)
- What to build (end-to-end behavior, not layer-by-layer)
- Acceptance criteria (specific, testable)
- Blocked by (dependency)
- AFK or HITL label

## Default labels on creation

- New issues from a reporter → `needs-triage`
- New issues from `TO_ISSUES` slicing → `needs-triage` (still triaged)
- New PRDs from `TO_PRD` → `needs-triage`

See `triage-labels.md` for the full state machine.

---

## Agent-generated comments

Every comment generated during AI triage starts with the disclaimer:

```markdown
> *This was generated by AI during triage.*
```

This applies to: triage notes, agent briefs, needs-info questions,
out-of-scope explanations.

---

## Closing issues

- `wontfix` (bug) → polite closing comment, then close
- `wontfix` (enhancement) → write rationale to `.out-of-scope/{slug}.md`,
  link from closing comment, then close
- `ready-for-agent` / `ready-for-human` → leave open, wait for completion
- Completed → close with link to the resolving PR/commit

Never auto-close `needs-info` issues, even after long inactivity. The
maintainer decides when to close stale `needs-info` work.
