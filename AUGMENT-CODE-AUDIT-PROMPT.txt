## SYSTEM IDENTITY

SEO-AI-REGENT is an SEO SaaS application competing with Surfer SEO, Clearscope, and Frase. Its core modules are:

1. **SERP Analyzer** — fetch top-ranking results and derive actionable optimization patterns
2. **Content Editor** — real-time scoring editor with term and structure guidance
3. **Keyword Research** — clustering, gaps, and opportunity discovery
4. **AI Writer** — outline and draft generation aligned to SERP patterns
5. **Audit** — URL-based scoring against live competitors
6. **GEO Score** — AI-search visibility scoring for ChatGPT / Perplexity / Gemini

The core promise is:

**“Content scoring for Google AND AI search.”**

The audit must verify that the codebase actually delivers this promise, not merely that the UI has screens for it.

---

## GODMYTHOS v9 OPERATING RULES

1. **Truth over theater** — do not mark a feature complete unless it works end-to-end.
2. **Evidence over approximation** — if a value can be derived, extracted, measured, or tested, do that instead of guessing.
3. **Fix before praise** — do not celebrate architecture or polish until failure paths, correctness, and trust are verified.
4. **Score transparency is mandatory** — any scoring system must explain inputs, weights, movement, and recommended actions.
5. **Competitive credibility matters** — the product must withstand comparison to Surfer, Clearscope, and Frase in both UX and code robustness.
6. **Design discipline is a product requirement** — visual sloppiness, token drift, and generic SaaS template patterns are audit failures.
7. **No dead surfaces** — every button, action, panel, and route must be connected to real logic or removed.
8. **No black-box billing** — Stripe state, entitlements, and plan transitions must be explicit, verified, and reversible.
9. **No silent failure** — every catch path must log structured context and preserve system integrity.
10. **Ship-readiness is binary** — unresolved critical paths mean “not ready,” even if most of the app works.

## REQUIRED EXECUTION STYLE FOR AUGMENT CODE

When operating inside the repo:

- Read code before changing it.

## PHASE 2 — PRODUCT WORKFLOW CORRECTNESS

Audit whether the app’s core workflows work end-to-end.

### 2A. SERP Analyzer
Verify:
- keyword submission works
- external SERP provider is correctly called
- retries / rate limiting / timeouts are handled
- response parsing is robust
- headings/entities/term signals are extracted correctly
- cache behavior is correct
- stale or partial SERP data does not silently poison scoring

Fix:
- invalid parser assumptions
- stale cache misuse
- weak normalization
- useless raw-data output with no synthesis
- broken loading or progress feedback

### 2B. Content Editor
Audit the editor as the revenue-critical feature.

Verify:
- loading an article works
- autosave works
- score recalculation is triggered correctly
- editor changes propagate to the scoring layer
- term suggestions map to real analysis output
- content score breakdown reflects actual scoring logic
- GEO score panel is distinct and coherent
- long-form content remains stable
- editor state survives refresh/navigation appropriately

If the editor is only cosmetically integrated with scoring, it fails.

### 2C. Audit Flow
Verify:
- URL fetch path works on real URLs
- HTML/content extraction is robust
- audit score uses the same scoring model as editor scoring where intended
- suggested fixes are prioritized
- error paths are usable
- audit results persist if the product claims persistence
- export/share path works if exposed in UI

### 2D. Keyword Research
Verify:
- clustering logic actually groups keywords meaningfully
- duplicate or near-duplicate keywords are handled
- search intent, difficulty, overlap, and gaps are coherent if shown
- outputs connect to article creation or audit workflows

### 2E. AI Writer
Verify:
- outline generation works
- draft generation works
- generated content returns to editor correctly
- prompt-response parsing is safe
- retries and rate limits are handled
- users can edit generated content without breaking the scoring pipeline

### 2F. GEO Score
Audit aggressively:
- inputs are real
- logic is not duplicated sloppily from Content Score
- sub-dimensions exist and are understandable
- recommendations map to specific GEO dimensions
- output is explainable enough to earn trust

If GEO Score feels invented, arbitrary, or disconnected from displayed recommendations, treat as a critical product failure.

## PHASE 3 — SCORING ENGINE AUDIT

This phase is non-negotiable. The scoring system is the trust core.

### 3A. Content Score Verification
Trace:
- input
- normalization
- signal calculation
- weighting
- aggregation
- persistence
- rendering

Confirm alignment with the intended weighted multi-signal model:
- termFrequency
- entityCoverage
- headingStructure
- wordCount
- readability
- internalLinks
- geoSignals contribution

### 3B. GEO Score Verification
Trace:
- entityAuthority
- factualDensity
- answerFormat
- sourceCredibility
- freshness

Confirm:
- weights are implemented
- no hidden defaults flatten weak data into fake confidence
- errors do not silently degrade to misleading “valid” scores

### 3C. Explainability Requirement
Every score surface must support:
- what changed
- why it changed
- which signal moved
- what the user should do next
- ideally estimated impact of the recommended fix

If the code calculates score but the UI cannot explain it, the feature is incomplete.

### 3D. Adversarial Scenarios
Test:
- keyword-stuffed garbage content
- excellent structure but missing entities
- extremely long content
- very short content
- malformed HTML
- sparse SERP
- duplicate headings
- missing title
- external API partial failure
- great SEO score but weak GEO score
- good GEO score but weak traditional SEO structure

The system must behave sensibly.
